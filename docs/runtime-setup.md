# MCP-1C Runtime — выполнение запросов на живой базе 1С

Runtime-слой превращает MCP-1C из «парсера XML» в **двунаправленный** ассистент: Claude может выполнить запрос на боевой базе, проверить гипотезу, посмотреть содержимое справочника. Аналогов на рынке 1С + LLM нет.

## Архитектура

```
┌─────────────────────┐       Bearer-token HTTPS        ┌──────────────────────┐
│   Claude Code/Desktop│ ──────────────────────────────▶│  MCP-1C Python server│
└─────────────────────┘                                 │  RuntimeClient       │
                                                        └──────────┬───────────┘
                                                                   │ HTTP/JSON
                                                                   ▼
                                              ┌─────────────────────────────────┐
                                              │  1С: Предприятие — стенд        │
                                              │  Расширение MCPBridge.cfe       │
                                              │  HTTP-сервис /hs/mcp/...        │
                                              └─────────────────────────────────┘
```

Python-сторона — `mcp_1c.engines.runtime`. 1С-сторона — отдельное расширение конфигурации, которое **вы** устанавливаете на свою тестовую базу.

## 1С-сторона: контракт MCPBridge

Расширение должно содержать ровно один HTTP-сервис с пятью операциями. Корневой URL — `/hs/mcp` (имя HTTP-сервиса свободное, путь на сервере — фиксированный).

### Аутентификация

Все запросы должны нести заголовок:

```
Authorization: Bearer <токен>
```

Токен — константа в расширении (`Константа.МСПB_АутентификационныйТокен`). Запрос без или с неверным токеном — `401`.

### Эндпоинты

#### `GET /status`

Проверка связи. Ответ:

```json
{
  "version": "1.0.0",
  "configuration": "УТ 11.5",
  "user": "АДМИН",
  "read_only_mode": false,
  "extension_version": "1.0.0"
}
```

#### `POST /query`

Выполнить запрос языком запросов 1С.

Запрос:

```json
{
  "query": "ВЫБРАТЬ Код, Наименование ИЗ Справочник.Номенклатура ГДЕ Код = &Код",
  "parameters": {"Код": "00001"}
}
```

Ответ:

```json
{
  "rows": [{"Код": "00001", "Наименование": "Товар"}],
  "row_count": 1,
  "duration_ms": 12
}
```

Реализация на стороне 1С обязана:
- ограничить максимальное количество строк (например, 1000) и срезать с предупреждением;
- запретить `ИЗМЕНИТЬ` / `ОБЪЕДИНИТЬ` / `УНИЧТОЖИТЬ` (это не язык запросов, но проверка `Найти(`УДАЛИТЬ`)` лишней не будет — на случай тонкого SQL-injection через текст);
- логировать запросы в журнал регистрации (событие `MCPBridge.Query`).

#### `POST /eval`

Выполнить BSL-фрагмент в безопасном контексте.

Запрос:

```json
{
  "fragment": "Возврат ТекущаяДатаСеанса();",
  "allow_writes": false
}
```

Ответ:

```json
{
  "result": "2026-04-30T22:00:00",
  "type": "Дата",
  "duration_ms": 1
}
```

Если `allow_writes=false`, расширение оборачивает `Выполнить` в проверку: запрещены `Записать()`, `Удалить()`, `Провести()` и аналогичные. Реализация — через предкомпилированный whitelist токенов или sandbox-объект (`БезопаснаяСреда` / `БезопасныйРежим` платформы).

#### `GET /data?type=...&name=...&guid=...`

Получить объект по ссылке. Возвращает все реквизиты и табличные части в JSON.

#### `POST /method`

Вызвать экспортную процедуру общего модуля.

Запрос:

```json
{
  "module": "ОбщегоНазначения",
  "method": "ТекущаяДатаПользователя",
  "arguments": []
}
```

1С-сторона должна валидировать `module` против whitelist в константе расширения — иначе любой вызывающий получит произвольный код.

### Минимальная реализация (.cfe)

Готовое расширение пока не входит в репозиторий — клиентский код Python готов и описан, но 1С-сторона требует развёртывания на стенде. Ожидаемый объём — один HTTP-сервис, одна константа (токен), одна константа (whitelist модулей), один общий модуль `МСПB_Сервер` с пятью функциями. Скелет планируется опубликовать отдельно (см. roadmap, Phase 6).

## Python-сторона: настройка

Переменные окружения:

| Variable | Назначение |
|----------|------------|
| `MCP_RUNTIME_BASE_URL` | Например, `https://1c.example.com/myinfobase/hs/mcp` |
| `MCP_RUNTIME_TOKEN` | Bearer-токен (соответствует константе в расширении) |
| `MCP_RUNTIME_TIMEOUT` | Таймаут запроса в секундах (по умолчанию 30) |
| `MCP_RUNTIME_RW` | `true`/`false`. По умолчанию `false` — write-операции запрещены |

Без этих переменных runtime-tools (`runtime-*`) видны в `list_tools`, но при вызове отдают `RUNTIME_NOT_CONFIGURED`.

## Безопасность

1. **Никогда не подключайте боевую базу** на старте — используйте копию.
2. Bearer-токен — не публикуйте в репозитории и не передавайте в логи.
3. Held-write-режим (`MCP_RUNTIME_RW=false`) — дефолт для всех экспериментов.
4. На стороне 1С обязательно держите whitelist разрешённых модулей в `runtime-method`.
5. Логируйте каждый вызов через журнал регистрации (`Событие = "MCPBridge.*"`).

## Skills поверх runtime (после реализации .cfe)

После Phase 6 будут доступны:

- `/1c-test-query` — пользователь даёт запрос, MCP выполняет на живой базе и показывает результат.
- `/1c-debug-data` — для отладки документа: получить его данные, регистры, движения.
- `/1c-explore-data` — посмотреть содержимое справочника.

## Проверка связи

```
runtime-status
```

Должно вернуть `{ "status": { ... }, "allow_writes": false }` или структурированную ошибку.
