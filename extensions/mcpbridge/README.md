# MCPBridge.cfe — мост из MCP-сервера в работающую 1С

Расширение конфигурации, которое **MCP-серверу** даёт три HTTP-метода в живую информационную базу 1С:

- `GET /mcpbridge/status` — health и информация о сервере 1С
- `POST /mcpbridge/query` — выполнить запрос на языке 1С (read-only)
- `POST /mcpbridge/eval` — выполнить произвольный код в транзакции с откатом

Это — **скелет**, не готовая поставка. Файлы здесь — это содержимое модулей, которые 1С-разработчик должен импортировать в Конфигуратор / EDT при создании расширения. Тестовая обвязка и форма авторизации остаются за разработчиком.

---

## Зачем оно нужно

MCP-сервер (этот проект, `mcp-1c`) умеет читать XML-выгрузку и понимать конфигурацию **статически**. Но для проверки гипотез типа «а как поведёт себя этот регистр расчётов в декабре, если...» нужен **живой** ответ от 1С — то есть метод, который выполнит запрос и вернёт строки.

`MCPBridge` — это и есть тот метод. Расширение публикуется **на тестовом контуре** или **на копии продуктивной базы**, где сервер 1С имеет открытый HTTP-порт, и MCP-сервер ходит туда за фактами.

---

## Что в этой папке

- `HTTPServiceModule.bsl` — модуль HTTP-сервиса. Маршрутизирует входящие запросы в общий модуль.
- `CommonModuleMCPBridge.bsl` — общий модуль с бизнес-логикой (запрос, eval, JSON-сериализация).
- `README.md` — этот документ.

## Установка

### 1. Создать расширение в Конфигураторе

1. **Конфигурация → Открыть конфигурацию** (если ещё не открыта).
2. **Конфигурация → Расширения конфигурации → Добавить.** Имя: `МСПМост`. Назначение: `Дополнение`.
3. В дереве расширения создать:
   - **Общие → HTTP-сервисы → Добавить**, имя `MCPBridge`, корневой URL `mcpbridge`.
   - У HTTP-сервиса добавить **шаблон URL `/`**:
     - Метод `GET` → имя обработчика `StatusGet`
   - Добавить шаблон URL `/query`:
     - Метод `POST` → имя обработчика `QueryPost`
   - Добавить шаблон URL `/eval`:
     - Метод `POST` → имя обработчика `EvalPost`
4. **Общие → Общие модули → Добавить**, имя `МСПМост`. В свойствах включить флаги: «Сервер», «Внешнее соединение». Снять «Клиент (управляемое приложение)», «Клиент (обычное приложение)», «Глобальный», «Привилегированный режим».

### 2. Подставить код

1. Открыть **модуль HTTP-сервиса MCPBridge** → вставить содержимое `HTTPServiceModule.bsl`.
2. Открыть **общий модуль МСПМост** → вставить содержимое `CommonModuleMCPBridge.bsl`.
3. **Конфигурация → Обновить конфигурацию базы данных** (F7).

### 3. Опубликовать сервер на http

```
1cv8 publish-ws --conn-string "Srvr=...;Ref=..." \
                --name mcpbase \
                --webroot /var/www/1c \
                --enable-http-services
```

Альтернативно — через **Администрирование → Публикация на веб-сервере** в Конфигураторе.

После публикации сервис доступен по: `http://<host>/<webroot>/hs/mcpbridge/...`

### 4. Создать технического пользователя

В **Администрирование → Пользователи** создать пользователя `mcp-bridge` с минимальным набором ролей:
- `Запуск веб-клиента` или эквивалент,
- роли только на чтение нужных объектов (для `/query`),
- **никаких ролей на запись** при штатной эксплуатации (для `/eval` с `read_only=true`).

Аутентификация — Basic. Пароль хранить в секрет-стороне MCP-сервера (env `MCP_1C_BRIDGE_USER`, `MCP_1C_BRIDGE_PASSWORD`).

## Контракты HTTP

### `GET /mcpbridge/status`

```http
GET /mcpbase/hs/mcpbridge/ HTTP/1.1
Authorization: Basic <base64>
```

Ответ `200`:
```json
{
  "ok": true,
  "version": "8.3.24.1234",
  "locale": "ru_RU",
  "ts": "2026-05-01T15:23:00",
  "user": "mcp-bridge"
}
```

### `POST /mcpbridge/query`

```http
POST /mcpbase/hs/mcpbridge/query HTTP/1.1
Authorization: Basic <base64>
Content-Type: application/json

{
  "text": "ВЫБРАТЬ Ссылка, Наименование ИЗ Справочник.Контрагенты ГДЕ НЕ ПометкаУдаления",
  "params": {}
}
```

Ответ `200`:
```json
{
  "ok": true,
  "rows": [
    { "Ссылка": "Контрагент №1", "Наименование": "ООО Альфа" },
    { "Ссылка": "Контрагент №2", "Наименование": "ООО Бета" }
  ],
  "count": 2
}
```

При ошибке:
```json
{ "ok": false, "code": "QUERY_FAILED", "message": "..." }
```

### `POST /mcpbridge/eval`

```http
POST /mcpbase/hs/mcpbridge/eval HTTP/1.1
Authorization: Basic <base64>
Content-Type: application/json

{
  "code": "Дата = ТекущаяДатаСеанса()",
  "read_only": true
}
```

Ответ `200`:
```json
{
  "ok": true,
  "result": "2026-05-01T15:24:00",
  "messages": [],
  "committed": false
}
```

`read_only: true` (значение по умолчанию) **всегда** откатывает транзакцию. `read_only: false` фиксирует, но требует от MCP-сервера scope `execute:write` — без него запрос отклоняется ещё на стороне MCP, до похода в 1С.

---

## Защита

`POST /eval` — это де-факто право выполнять любой код от лица технического пользователя. Поэтому:

1. **Не публиковать MCPBridge на продуктовой базе.** Только тест/копия продуктива.
2. **Сервер 1С должен слушать HTTP только из локальной сети** (firewall / nginx upstream-only).
3. **Технический пользователь** имеет роли только на чтение. `read_only: false` физически не сможет ничего записать без записанных ролей.
4. **Basic-auth поверх TLS.** Никаких HTTP без шифрования.
5. **Аудит-лог** на стороне MCP уже пишется (`utils/audit.py`); можно завести отдельный лог 1С через `ЗаписьЖурналаРегистрации` в `ВыполнитьКод` если хотите trail на стороне 1С.

## Интеграция в MCP-сервер (TODO для python-кода)

В этом репо ещё **нет** клиента, который дёргает MCPBridge. Когда будете писать:

- env-переменные: `MCP_1C_BRIDGE_URL`, `MCP_1C_BRIDGE_USER`, `MCP_1C_BRIDGE_PASSWORD`
- модуль: `src/mcp_1c/engines/bridge/client.py` с aiohttp
- `BridgeStatusTool`, `BridgeQueryTool`, `BridgeEvalTool` в `src/mcp_1c/tools/bridge_tools.py` (унаследовать от `BaseTool`)
- Scope для `bridge-eval` — `execute:write` (см. `src/mcp_1c/auth/scopes.py`)
- При `read_only=true` использовать scope `execute:read` (по умолчанию)
- Регистрация — в `ToolRegistry._register_all_tools()`

Архитектурно ориентироваться на `tools/embedding_tools.py` как образец интеграции внешнего HTTP-сервиса.

## Известные ограничения

- **Закодировано как Выполнить("Результат = " + Код)** — поддерживается только одно выражение, не сценарий. Для многострочных вычислений нужна обвязка `ВыполнитьМодуль`/JIT-компиляция, добавляйте при необходимости.
- **Ссылки сериализуются как строка представления.** Если нужны UUID — заменить `СериализоватьЗначение` на свою реализацию.
- **Параметры запроса** — без декларации типов; всё, что не поддерживается JSON (даты, ссылки), нужно передавать в виде, который 1С распознает (`Дата("20260501")`).
- **HTTP/2 / gRPC** — нет; платформа умеет только HTTP/1.1.
